### 🔴 `rt5` — cost-ops (high)

**Concern.** The daemon fsyncs every event while the server indexes every event, putting disk durability on the hottest path.

**Consequence.** At the stated load, chunk events can dominate runtime cost and miss latency targets through fsync backpressure. The plan tests this late but does not define an operating mode that trades a tiny loss window for predictable throughput.

**Counter-proposal.** Use durability tiers: fsync lifecycle and terminal events immediately, group-commit chunk/progress events every 50 events or 100 ms, and expose queue depth plus fsync latency in `/api/health`.

**Trade-off.** Balanced mode may lose the last small batch of non-lifecycle output on power loss.

### 🔴 `rt3` — data (high)

**Concern.** Sweeper-only crash transitions update SQLite without producing any JSONL event.

**Consequence.** WebSocket backfill through `event_offsets` cannot replay the terminal crash event, while `/api/runs` reports the run as crashed. Exports and analytics that treat JSONL as canonical will disagree with the UI.

**Counter-proposal.** Add a `synthetic_events` table with server-assigned terminal events and union it with JSONL `event_offsets` in WS/API backfill.

**Trade-off.** Gives up the simpler claim that JSONL alone is the complete event contract.

### 🔴 `rt2` — reliability-distsys (high)

**Concern.** Pending rewrite recovery is split between SQLite state and a host-side journal the server cannot read.

**Consequence.** A crash between pre-rename and update-mtime can leave `rewrite_pending=1` while the tailer skips the file indefinitely. The recovery story depends on journal intent, but the journal is not part of the container mounts.

**Counter-proposal.** Make SQLite the sole rewrite transaction log with `rewrite_txn_id`, `rewrite_state`, `target_size_bytes`, and `target_mtime_ns`; recover by statting the file against DB state at server startup.

**Trade-off.** More schema and endpoint complexity, and less independent recovery by the prune CLI while the server is down.

### 🔴 `rt1` — security-trust (high)

**Concern.** Any same-UID local process can connect to the 0600 writer socket and forge observability events.

**Consequence.** The read-only container mount prevents container-side tampering, but the host side remains trusted wholesale. A compromised helper or unrelated user process could fabricate lifecycle events, hide findings, or flood a run before indexing.

**Counter-proposal.** Run the writer daemon and spool under a dedicated `starkobs` OS user managed by launchd, and require dispatchers to submit through an authenticated client helper that grants scoped per-run capabilities.

**Trade-off.** Adds local setup complexity and makes ad hoc emit/debug flows less convenient.
