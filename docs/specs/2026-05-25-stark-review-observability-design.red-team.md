## Synthesis

The main tension is between full-fidelity live logging for reliability/product value and the security/cost blast radius of persisting every token stream for 30 days. A second tension is self-contained Docker operation versus trust-boundary cleanliness: making the container own retention simplifies setup, but gives the web-facing process write power over the evidence spool.

## Findings

### 🟣 `rt2` — reliability-distsys (critical)

**Concern.** The liveness model cannot meet the stated 60-second crashed criterion when the dispatcher dies before writing `run_end`.

**Consequence.** The server has no host process signal, so orphaned runs are only inferred after 1800 seconds. A dead 90-minute review can therefore look merely stalled for far longer than the success criteria allow.

**Counter-proposal.** Add a run-level heartbeat plus `parent_pid` in `meta.json`; mark non-terminal subagents crashed when the run heartbeat expires after the 300-second activity window plus 60 seconds, with host-sleep detection to avoid false positives.

**Trade-off.** Adds periodic writes and needs explicit handling for laptop sleep/resume.

### 🔴 `rt5` — cost-ops (high)

**Concern.** Full stdout/stderr capture has no storage quota or pressure policy.

**Consequence.** At the stated load-test rate, one 90-minute run can exceed a gigabyte before JSON overhead, and 30 days of repeated runs can fill the user's home or Docker disk. Observability then fails exactly when logs are needed.

**Counter-proposal.** Add total and per-run byte budgets, low-disk preflight in `startRun`, health/UI disk alarms, and pressure retention that prunes oldest terminal runs or truncates token chunks while preserving lifecycle and progress events.

**Trade-off.** Some historical token streams may be truncated or expire before 30 days.

### 🔴 `rt3` — data (high)

**Concern.** The design exposes `from_seq` chunk replay but stores no per-chunk byte offset index.

**Consequence.** Historical log reads may require scanning large rotated JSONL files to find the requested sequence. The 2-second replay target will degrade as 30-day history accumulates.

**Counter-proposal.** Add a `chunk_offsets` table keyed by `(run_id, subagent_id, seq)` with stream, rotation index, byte start, byte end, timestamp, and encoding; if size is a concern, store segment offsets every fixed number of chunks.

**Trade-off.** More SQLite write volume and a larger index.

### 🔴 `rt4` — product-dx (high)

**Concern.** The first-run auth path is contradictory: the API requires a bearer token, the container generates it at runtime, and the runtime config endpoint does not return it.

**Consequence.** A user can start the stack and open the UI but still get unauthorized API calls unless they manually inspect Docker logs and perform an unspecified setup step. That breaks the core emergency use case of checking a long-running review quickly.

**Counter-proposal.** Add a `stark-observability open` helper that reads the token via Docker, opens localhost with a one-time bootstrap code, and exchanges it for an HttpOnly same-origin session cookie.

**Trade-off.** Adds a small bootstrap command and session state instead of a purely static UI.

### 🔴 `rt1` — security-trust (high)

**Concern.** The web-serving container is also a read-write principal over the host spool it is supposed to observe.

**Consequence.** If the server, a dependency, or the bearer token is compromised, the attacker can tamper with or delete run evidence. This undermines the append-only audit contract and expands the blast radius beyond viewing logs.

**Counter-proposal.** Bind-mount the spool read-only into the container; move deletion to a host-side `observability prune` command or launchd job, and keep synthetic crashed states in SQLite rather than writing them back to the spool.

**Trade-off.** Retention is no longer fully self-contained in Docker and needs a host cleanup path.

**Summary.** Preserve the append-only observability contract, but narrow the Docker trust boundary, make liveness independently detectable, and add the missing indexes, bootstrap flow, and storage pressure controls needed for the stated latency and emergency-use goals.

### 1. Separate spool ownership

**Addresses:** `rt1`
**Sections touched:** `Architecture`, `§5 Docker server`, `§10 Retention`, `Security`, `Deployment`

**Rationale.** Mount the event spool read-only in the web container, move deletion to a host-side prune path, and keep inferred crashed states in SQLite. This preserves Docker-based viewing while preventing the web-facing process from modifying or deleting the host evidence it observes.

**New trade-off.** Retention is no longer fully self-contained in Docker and requires a host cleanup command or launchd job.

### 2. Add run-level liveness

**Addresses:** `rt2`
**Sections touched:** `§2 Spool layout`, `§4 Liveness model`, `§5 Docker server`, `§6 SQLite index schema`, `Testing`

**Rationale.** Introduce a run heartbeat and parent process identity so the server can distinguish a dead dispatcher from a quiet sub-agent. Crash inference should use expired run heartbeat plus the sub-agent activity window, with laptop sleep/resume detection called out as part of the design.

**New trade-off.** Periodic run-heartbeat writes add overhead and require explicit handling for host sleep to avoid false crash states.

### 3. Index chunk replay positions

**Addresses:** `rt3`
**Sections touched:** `§3 Rotation`, `§5 Docker server`, `§6 SQLite index schema`, `§7 HTTP API`, `Load / latency`

**Rationale.** Add a chunk-position index for token stream events so historical `from_seq` reads can seek directly into rotated JSONL files instead of scanning large logs. The design can choose exact per-chunk offsets or coarser segment offsets, but the replay contract needs an indexed path.

**New trade-off.** SQLite write volume and index size increase during high-output runs.

### 4. Make bootstrap explicit

**Addresses:** `rt4`
**Sections touched:** `§7 HTTP API`, `§8 WebSocket protocol`, `Security`, `Deployment`

**Rationale.** Replace the contradictory static-token UI story with a first-class local open flow: a helper retrieves or provisions the token, opens localhost with a short-lived bootstrap code, and exchanges it for a same-origin HttpOnly session. HTTP and WebSocket auth can then depend on the session without requiring the operator to inspect Docker logs during an incident.

**New trade-off.** The stack gains session state and a small helper command instead of being a purely static browser UI.

### 5. Enforce storage budgets

**Addresses:** `rt5`
**Sections touched:** `§1 Emit library`, `§2 Spool layout`, `§7 HTTP API`, `§10 Retention`, `Testing`

**Rationale.** Define per-run and total spool byte budgets, add low-disk checks at run start, surface disk pressure in health/UI, and specify pressure retention that preserves lifecycle and structured progress while pruning or truncating older token chunks first.

**New trade-off.** Some historical token streams may be truncated or expire before the nominal 30-day retention window.
