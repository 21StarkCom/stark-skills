This skill is a guarded dependency-upgrade pipeline, not a blind version bump. Its core contract is: discovery → registry-backed research → compatibility triage → in-place edits → exact-version verification → explicit report.

Decision Surface

Safe

Patch or minor update, no known breaking changes, runtime support preserved, and exact version confirmed on an official registry.

Review

Usually a major bump. Version may be valid, but migration work, codemods, or API changes are likely. The skill surfaces concrete migration commands when available.

Blocked

Conflict with project runtime, peer dependency mismatch, unsupported latest release, or Docker/software version mismatch where the image tag does not actually exist.

Internal Policies

Registry Truth Over Guesswork

Every candidate version is sourced from official registries or release pages. The skill assumes hallucination risk is real and adds a mandatory second verification pass after edits.

LTS-Biased Runtime Selection

Node and other runtime base images prioritize Active LTS and broad ecosystem compatibility instead of simply selecting the numerically newest release.

Constraint-Style Preservation

Manifest syntax is respected. Existing range semantics remain intact so the upgrade changes the intended minimum or target version, not the package management strategy.

Docker Tag Skepticism

Docker tags are treated as a separate truth source. A software release does not imply a matching Docker Hub tag, especially for database images and CUDA variants.

Extension Points

New Manifest Parsers

Add support for another ecosystem by emitting the same inventory record shape: package identifier, current version, file location, and ecosystem label.

Additional Registry Adapters

Private registries or internal package indexes can slot into the research phase as new official-source providers as long as they return stable version and evidence metadata.

Stricter Policy Gates

Teams can tighten the decision matrix with extra blocked conditions, such as minimum test coverage, changelog review requirements, or allowlists for major upgrades.

Failure Modes And Recovery
Failure Mode Detection Point Recovery
Fabricated or nonexistent package version Verification exact-version search Search again broadly, replace with confirmed latest stable, then re-verify
Docker tag does not exist Research or verification against Docker Hub Select closest valid published tag or revert original image reference
Major upgrade breaks runtime support Compatibility analysis Mark Blocked or Review, keep current version, surface migration path
Cross-file version drift Consistency check after update Align duplicate dependency references across manifests and docs
Private package not found publicly Research phase Skip and report as private/unverified instead of inventing a public version
Why Internals Should Care

Single-Pass Multi-Ecosystem Audit

The skill operates like a repository-wide dependency control plane. Python, Node, Docker, Go, Rust, Java, Ruby, .NET, CI, infra, and docs all feed one evaluation path.

Evidence-Carrying Updates

The final report is not just a diff summary. It carries the proof needed to trust each changed version and calls out where human review is still required.

Operationally Useful Output

Major bumps are translated into action items like migration guides and codemods, turning a version audit into a workable change plan for follow-up implementation.
