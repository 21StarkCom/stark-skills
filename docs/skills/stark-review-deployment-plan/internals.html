10 Required Failure Vectors
A
Partial-Failure Trap
Dual-write / migration cutovers. Simulate both directions: A✓/B✗ and B✓/A✗. Require anti-split-brain gate.
B
Imperative Idempotency
Identify non-idempotent imperative commands (create, deploy, execute) in CI/CD and scripts.
C
Blank-Slate IaC
Assume empty project/account. Flag hidden assumptions about pre-existing service agents, IAM, networking.
D
Dependency Sequencing
Validate infra + runtime dependencies across phase boundaries. Call out phase-order breakages.
E
Reality Drift
Non-IaC resources: define authoritative owner, drift correction mechanism, or detection + manual remediation.
F
Command Contract
Validate CLI/API usage and flags. Mark UNCERTAIN if behavior is ambiguous; never fabricate.
G
Pre-Cutover Gate
Dual-write migrations require explicit final delta sync or write freeze before read cutover. FAIL if missing.
H
API Prerequisite Matrix
Map each phase capability to required cloud APIs/services. FAIL if any required API is implied but undeclared in IaC.
I
Identity Lifecycle
Each runtime identity needs explicit creation → IAM binding → deploy-time attachment. FAIL if incomplete.
J
Evidence Strictness
Missing evidence = FAIL or DEFERRED with explanation. No pass by omission.
Severity & Confidence Rubrics

Severity Classification

HIGH Execution blocker, data loss/divergence, security break, guaranteed drift
MEDIUM Likely operational issue with workaround
LOW Non-blocking improvement

Confidence Classification

HIGH Verifiable from document text alone
MEDIUM Requires assumption about runtime/external system
LOW Uncertain about underlying API/tool behavior

Output Structure — 5 Required Sections
# Section Content Ordering
1 Findings Material flaws: severity, location, failure scenario, why plan fails, fix, confidence High → Medium → Low
2 Dependency Violations Table Prerequisite · dependent step · what breaks · fix By dependency chain
3 Ownership & Drift Table Every resource in the plan: owner, drift risk, correction mechanism All resources, not just problematic
4 API Prerequisite Matrix Capability/phase → required API → IaC enablement → evidence → status By phase
5 Coverage Matrix Vectors A–J: PASS / FAIL / DEFERRED with one-line evidence A through J
Hard Constraints & Guards

No Fabrication

Never invent facts, file contents, command behavior, or documentation URLs. If uncertain about command/API semantics, mark UNCERTAIN and explain the risk.

DEFERRED Protocol

If codebase/scripts are inaccessible, explicitly mark those checks as DEFERRED. List exactly what was not verifiable.

No Pass by Omission

Every vector requires concrete evidence to pass. If evidence is missing, use FAIL or DEFERRED — never skip silently.

False-Positive Guard

Do not report stylistic preferences or theoretical edge cases. Every High finding must include a concrete production failure sequence.

Quality Bar

No generic advice. No "consider X." PASS needs evidence. DEFERRED needs reason. Every finding must be actionable.

Traceability

Use exact line numbers when available. Otherwise quote exact section text or snippet from the plan document.

Internal Data Flow

Input Resolution

The skill accepts three input forms: inline text (already in context), file path (resolved via Read tool), or url(). If none provided, the skill halts and asks the user for a plan document before proceeding.

Multi-Pass Accumulation

Each pass builds on the previous. Pass 1 creates the structural model. Pass 2 identifies contradictions against that model. Pass 3 validates against external state. Pass 4 runs each vector against the accumulated understanding. Findings from earlier passes inform later evaluation.

Vector Independence

The 10 vectors (A–J) are evaluated independently but may share evidence. A finding in vector A (partial-failure) often correlates with vector G (pre-cutover gate). The coverage matrix captures the independent verdict for each.
