Quick Start

Invoke

Type /stark-review-deployment-plan followed by a file path, URL, or paste inline.

What It Does

Runs a 4-pass adversarial review across 10 failure vectors, producing findings ranked by blast radius.

What You Get

Structured report: findings, dependency table, drift table, API matrix, and coverage matrix.

How to Invoke
> /stark-review-deployment-plan docs/migration-plan.md
> /stark-review-deployment-plan https://confluence.example.com/deploy-plan
> "review this deployment plan" — then paste the plan inline

10 Failure Vectors Evaluated
A
Partial-Failure Trap
Dual-write cutovers: simulates A-succeeds/B-fails and vice versa. Requires anti-split-brain gate.
B
Imperative Idempotency
Identifies non-idempotent CI/CD commands (create, deploy, execute) that aren't upsert-safe.
C
Blank-Slate IaC Completeness
Assumes empty project: checks APIs enabled in IaC, flags hidden pre-existing assumptions.
D
Dependency Sequencing
Cross-boundary infra + runtime dependency ordering. Catches phase-order breakages.
E
Reality Drift
Non-IaC resources: authoritative owner, drift correction or detection + manual remediation.
F
Command Contract Validation
Validates CLI/API flags and usage. Marks UNCERTAIN if behavior can't be confirmed.
G
Pre-Cutover Gate
Dual-write migrations must have explicit final delta sync or write freeze before read cutover.
H
API Prerequisite Matrix
Maps each phase to required cloud APIs. Fails if any API is implied but undeclared in IaC.
I
Runtime Identity Lifecycle
Every runtime identity needs explicit creation, IAM bindings, and deploy-time attachment.
J
Evidence Strictness
No pass-by-omission. Every vector requires concrete evidence. Missing = FAIL or DEFERRED.
Output Structure

The review produces five structured sections, always in this order:

1. Findings (High → Medium → Low)

Each finding includes: Severity location, concrete failure scenario, why the plan fails, exact technical fix, and confidence level.

2. Dependency Violations Table

Prerequisite → dependent step mapping. Shows what breaks if a dependency is missing and the fix.

3. Ownership & Drift Table

Every resource in the plan: owner, drift risk, and correction/detection mechanism.

4. API Prerequisite Matrix

Capability/phase → required API/service → explicitly enabled in IaC? → evidence → status.

5. Coverage Matrix

Vectors A–J, each marked PASS FAIL or DEFERRED with one-line evidence.

The report always ends with: Remaining blockers before execution: <count>

Severity & Confidence

Severity Levels

HIGH Execution blocker, data loss, security break, guaranteed drift
MEDIUM Likely operational issue with workaround
LOW Non-blocking improvement

Confidence Levels

High — Verifiable from document text alone
Medium — Requires runtime/external system assumption
Low — Uncertain about underlying API/tool behavior

Common Workflows

Pre-Execution Gate

Run before executing any migration or deployment plan. Fix all High-severity findings before proceeding. Use the blocker count as your go/no-go signal.

Plan Iteration

After receiving findings, update the plan and re-run the review. Each pass should reduce the blocker count toward zero.

Casual Sanity Check

Paste a plan and say "does this look right?" or "poke holes in this" — the skill activates automatically and gives a full adversarial review.

Quality Guarantees

No Fabrication

Never fabricates command behavior or documentation URLs. Marks uncertain items as UNCERTAIN.

No False Positives

No stylistic preferences or theoretical edge cases. Every High finding includes a concrete production failure sequence.

Full Coverage

All 10 vectors are always evaluated. No pass-by-omission — missing evidence is explicitly marked FAIL or DEFERRED.
