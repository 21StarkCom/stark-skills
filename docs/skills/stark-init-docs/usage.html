usage

stark-init-docs

Scaffold, backfill, upgrade, or clean a standardized developer-docs layout in any Git repository
Best for repos that need a predictable docs/ structure, generated onboarding guides, ADRs, PR-backed specs, or a safe way to retire the scaffold later.

How You Invoke It

Use /stark-init-docs with one or more flags. If you pass nothing, the skill prints the available modes and waits for your choice before changing the repo.

What You Provide

A Git repo, access to the template directory, and optionally gh for merged PR history. The skill reads existing docs, repo configs, and commit history to decide what to generate.

What You Get

Standardized directories, MkDocs config, CODEOWNERS, generated guides, conservative ADRs, spec stubs for major merged features, or a cleanup pass that preserves user content.

Mode Guide
Template

--template

Bootstrap the standard doc skeleton without overwriting existing files.

  • Creates docs/specs, plans, adr, guides, reference, and architecture
  • Copies templates into mkdocs.yml, docs/index.md, PR template, staleness config, and ADR template
  • Creates .github/CODEOWNERS only when no CODEOWNERS file exists
  • Commits only if something was actually added
docs/index.md mkdocs.yml .doc-staleness.yml docs/adr/0000-template.md
Backfill

--backfill

Generate useful docs from the repo itself, using actual history and config instead of placeholders.

  • Runs --template first
  • Reads commit history, tries merged PR history through gh, and inspects language/build/deploy files
  • Generates conservative ADRs, up to 10 meaningful feature specs, and setup/development/deployment guides
  • Updates mkdocs.yml nav to include generated content
If gh is missing or fails, the skill warns and continues with commits and codebase analysis only.
Upgrade

--upgrade

Move scattered Markdown into a standard layout and fix the repo around it.

  • Scans tracked Markdown outside docs/ while excluding common noise like node_modules and root README.md
  • Classifies each file as spec, ADR, guide, reference, or architecture
  • Moves files with git mv, resolves naming collisions, then updates internal links and image paths
  • Refreshes MkDocs nav and runs --template afterward to fill missing pieces
Clean

--clean

Remove only scaffold assets, not authored documentation.

  • Always asks for confirmation before deleting anything
  • Deletes template files such as mkdocs.yml, .doc-staleness.yml, PR template, and the ADR template
  • Deletes generated .github/CODEOWNERS only when it still looks template-owned
  • Removes empty directories, then commits only if files were actually removed
Never deletes user-authored docs, README.md, CHANGELOG.md, or files with meaningful later history.

Common Workflows

New repo, no docs yet

Run /stark-init-docs --template. This is the lowest-risk path when you only need the folder structure, base config, and starter templates without inferred content.

Existing repo, docs are scattered or missing

Run /stark-init-docs --upgrade --backfill. First standardize old Markdown into the right folders, then generate missing ADRs, specs, and guides from the repo’s real history.

Repo already standardized, now enrich it

Run /stark-init-docs --backfill. This is useful when the folder structure exists but you still want generated onboarding and architecture docs wired into MkDocs navigation.

Remove the scaffold safely

Run /stark-init-docs --clean and confirm when prompted. The mode is designed to leave real docs in place and only strip the scaffold’s template-owned assets.

Guardrails And Failure Cases

Templates directory missing

The skill stops with: templates not found at ~/.claude/code-review/standards/templates/. Install the template set first.

Not in a Git repository

The skill errors immediately. All four modes assume Git is present because they read history, move files through Git, and create commits.

PR history unavailable

Backfill still works. It warns that PR history could not be fetched and falls back to commit history plus codebase analysis.

Observability output

Expect metrics on executed modes, files created/skipped/moved/deleted, generated ADR/spec/guide counts, and internal links updated during upgrade.

Practical rule of thumb: choose --template when you want structure only, --backfill when the repo can explain itself, --upgrade when docs are scattered, and --clean only when you explicitly want the scaffold gone but not the authored content.
