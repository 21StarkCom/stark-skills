stark-autopilot

Autonomous multi-agent implementation with tournament-per-step
USAGE • end-user focused

Use this when you want a substantial feature built end-to-end by Claude, Codex, and Gemini competing in parallel on every step, with the best result merged before the next step begins.

Best for major features 3 agents per step Tournament scoring Commits as it goes

How You Invoke It

Pass either an inline build request or a markdown plan file. Add a test command when you want each tournament to score test results as part of the winner selection.

What It Produces

Step-by-step implementation commits, tournament scorecards, per-step winners, saved run history, and optionally a PR summary comment.

What Makes It Different

Instead of one agent taking one pass, all three agents compete on every implementation step and the strongest solution becomes the base for the next round.

Arguments
Argument What it means When to use it
<plan-or-prompt> Path to a markdown implementation plan, or an inline task description. Always. This is the thing autopilot will build.
--test-command CMD Command to run after each step, such as npm test or pytest. Use when tests should influence the tournament winner.
--agents LIST Comma-separated agent IDs. Default is claude,codex,gemini. Use when limiting the field or swapping agent mix.
--timeout N Per-agent timeout in seconds. Default is 900. Raise it for bigger steps; lower it for tighter budget and faster turnover.
--dry-run Prints the plan and stops before execution. Use to confirm steps, agent list, and detected test command.
Common Workflows

1. Build from a plain request

Describe the feature in natural language. The skill decomposes the work into steps automatically, then runs the tournament loop across those steps.

Example: /stark-autopilot "Add SSO onboarding, admin approval, and user audit history"

2. Execute a prepared plan

Feed it a markdown plan from a design or planning workflow. Each phase or task heading becomes a step, preserving the structure you already approved.

Best when: architecture is already decided and you want implementation competition, not planning.

3. Run with tests as the referee

Attach a reliable test command so passing implementations receive a major score boost and weak but plausible diffs do not win on style alone.

Example: --test-command "pytest" or --test-command "npm test"
How Winners Are Chosen

Tournament scoring model

Every step produces up to three candidate diffs. Each candidate can earn up to 100 points total.

Tests
0 or 50
Semantic review
0-50
Winner
Best total
Semantic review always checks correctness, quality, completeness, integration, and simplicity.
If two agents tie, the workflow breaks ties using stronger implementation merit such as correctness.
The winning diff is applied to the main branch, committed, and becomes the starting point for the next step.
Observability You Will See

Task progress

The run creates explicit task entries such as setup, then one tournament task per implementation step so you can track where the process is spending time.

Timestamped logs

Expect timestamped status lines for dispatch, agent completion, test outcomes, winner selection, and apply/commit events.

Final metrics

Completion includes step count, tournament count, agent win rate, average scores, test pass rate, changed files, and commit count.

Failure Modes And Recovery
Failure What happens Recovery path
No input The run has nothing to build. It asks: What should I build?
1 of 3 agents succeeds No real tournament. Uses that output and warns reduced competition.
2 of 3 agents succeed Smaller field. Runs a reduced tournament and warns accordingly.
0 of 3 agents succeed Step cannot proceed safely. Aborts the step and asks how to continue.
All tests fail Test score contributes nothing. Falls back to semantic-only evaluation with a warning.
Diff fails to apply Winner cannot merge cleanly through patching. Copies files directly from the winner's worktree.
Worktree creation fails Parallel isolation is unavailable. Falls back to sequential same-branch execution.
Agent timeout That agent is disqualified for the step. Continue with remaining agents.
User aborts mid-run Temporary state could linger. Cleans up all worktrees before exit.
When To Reach For This Skill

Use it for

Large implementation asks, multi-file features, major refactors, or anything where multiple strong solution attempts are worth the extra runtime.

Do not use it for

Tiny one-line fixes, low-risk text edits, or tasks where the overhead of parallel tournaments would outweigh the value of deeper competition.

Trigger phrases

Common prompts include “autopilot,” “build this with all agents,” “tournament implementation,” or “let all 3 agents compete on building this.”

Mental Model

Think of it as iterative elimination

Each step is a mini championship round. Three agents start from the same repo state, produce competing implementations, and the strongest candidate becomes the new baseline. That pattern repeats until the full feature is done.

Practical effect: you are not betting on one agent being right on the first try. You are paying for competition, evaluation, and progressive convergence toward the best end-to-end implementation.
Best fit: high-value builds where quality matters more than speed
