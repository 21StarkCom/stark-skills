Phase 4 Synthesis — Internal Logic

1. Winner as Base

Start with the highest-scoring design's full structure and content. If a tie (within 0.5 points), both designs are loaded as co-bases.

2. Merge Superior Sections

For each section where a non-winning design scored higher on a specific dimension (e.g., security, extensibility), incorporate that design's approach into the corresponding section.

3. Address Flagged Weaknesses

Fix issues flagged by cross-reviewers. If both reviewers flagged the same weakness, it's a confirmed problem — don't carry it forward.

4. Quality Check

Verify: every requirement → design element, no section contradictions, security matches data sensitivity, open questions are genuine unknowns.

Extension Points

Agent List (--agents)

Default: claude,codex,gemini. Configurable via CLI flag. Adding an agent requires: a CLI binary on PATH, a prompts subdirectory in prompt-to-design/, and a corresponding GitHub App entry in github_app.py for review posting.

Prompts Directory

Generation and cross-review prompts live in the prompt-to-design/ directory under the prompts path. Each agent has its own variant. Edit these to change what the design documents focus on or how cross-reviews score.

Scoring Dimensions

Currently: Completeness, Clarity, Feasibility, Extensibility, Security. These are defined in the cross-review prompts — add/remove dimensions by editing the review prompt templates.

Output Path Logic

Default: docs/specs/YYYY-MM-DD-{slug}-design.md. Override with --output. The slug is derived from the input filename or auto-slugified from the inline prompt. Modify the path logic in Phase 5 of SKILL.md.

History & Metrics Integration

History written to ~/.claude/code-review/history/prompt-to-design/{slug}/. Files: generate.json, cross-review.json, synthesis.md, summary.md. Consumed by /stark-metrics and /stark-review-improvement for self-improvement loops.

Downstream Pipeline

Output design doc feeds into /stark-review-design for formal review, then /stark-design-to-plan for implementation planning. The full pipeline is automated by /stark-phase-execute.

Key Scripts & Dependencies

design_to_plan_dispatch.py

Orchestrator script. Modes: --mode generate (spawn 3 agents) and --mode cross-review (spawn 6 review agents). Uses ThreadPoolExecutor for parallelism. Returns JSON to stdout.

github_app.py

Multi-app GitHub auth. Generates JWT from macOS Keychain private key, exchanges for installation token. Supports stark-claude, stark-codex, stark-gemini apps. Used only when PR context detected.

prompt-to-design/

Per-agent prompt templates for both generation and cross-review. Agent preambles in agent.md, domain prompts in NN-domain.md. The prompts directory is the primary extension point.
