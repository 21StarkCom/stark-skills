What You Say

Invoke the skill directly or use natural language like “onboard project” or “bootstrap claude” in a directory that needs git, GitHub, and Claude Code setup.

What You Confirm

Repo org, name, visibility, description, and whether the detected project scan should be used to generate CLAUDE.md. You may also choose whether .claude/ stays private in .gitignore.

What You Get

A working git repo, GitHub remote, compliance topics, CODEOWNERS, all three Stark app connections, CLAUDE.md, and a local .claude/ directory ready for follow-up work.

Common User Workflows

1. Brand-New Local Folder

Use the skill in a directory that has source files but no git repo. It will initialize git, create the GitHub repo, wire app access, and generate Claude setup in one pass.

  • Best when starting a new codebase from scratch.
  • Defaults reduce questions: current folder, private repo, GetEvinced org.
  • Ends with a shareable repo URL and local Claude context.

2. Existing Git Repo, Missing GitHub Or Claude

If git is already initialized, the skill skips straight to the next missing phase. This is useful when a local project exists but remote setup or Claude onboarding has not been done yet.

  • No re-init of git.
  • No replacement of existing CODEOWNERS.
  • No overwrite of existing CLAUDE.md.

3. Repo Exists, Need Claude Setup Only

When origin already exists and the GitHub repo is connected, the skill extracts repo info and moves on. The remaining value is app verification plus CLAUDE.md and .claude/ setup.

  • Useful for legacy repos entering the Stark workflow.
  • Project scan gives users a chance to correct inferred tech stack.
  • Optionally keep .claude/ local-only via .gitignore.
Decision Points

What The Skill Will Not Overwrite

Two artifacts are protected by design: .github/CODEOWNERS and CLAUDE.md. If either already exists, the skill logs that fact and moves forward.

This keeps ownership data and prior Claude guidance stable unless a dedicated follow-up skill is used.

Where Humans Must Approve

The skill pauses before creating a GitHub repo and before generating CLAUDE.md. These are the two places where a wrong default would be expensive or noisy.

Everything else is operational: detect, verify, skip, or create as needed.

Failure And Recovery

gh Not Authenticated

The skill cannot create or inspect GitHub repos until GitHub CLI authentication is available. Recovery: run gh auth login and invoke the skill again.

Missing App Installation Scope

If app installation management returns a scope or 403 error, refresh auth with gh auth refresh -h github.com -s read:user, then rerun the skill.

App Access Cannot Be Added Automatically

Manual fallback exists for all three apps through the GetEvinced organization installation settings. Add the repository to each installation and rerun verification.

Repo Already Exists

If creation fails because the GitHub repo exists already, the recovery path is to add the remote and push instead of treating the run as failed.

No Language Detected

If the project scan cannot infer the stack, the skill asks the user to specify it instead of writing a weak generic CLAUDE.md.

Non-GetEvinced Repository

GitHub app wiring is a GetEvinced-only phase. When another org is used, the skill should skip app connection rather than forcing an invalid flow.
