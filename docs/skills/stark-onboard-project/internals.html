End-to-end bootstrap pipeline

This skill orchestrates a five-phase project bootstrap: local git state, remote GitHub provisioning, GitHub App access, Claude Code scaffolding, and final status reporting. Its core design goal is safe re-entry: each phase checks current state first, then either skips or executes.

Architecture Notes

Control Plane

The skill acts as a deterministic coordinator around local filesystem state, git metadata, GitHub repository APIs, GitHub App installations, and Claude-specific scaffolding. Its state machine is linear, but each phase begins with a read-only checkpoint so retries converge instead of duplicating work.

Data Flow

Inputs flow from the working directory, user confirmations, and GitHub identity. Derived values include repo name, project type, authenticated GitHub username, repo ID, and app installation visibility. Those values then feed artifact creation: .gitignore, .github/CODEOWNERS, CLAUDE.md, and .claude/.

Extension Points

Repo defaults can be changed by altering org/visibility prompts, project-type detection heuristics, topic assignment, GitHub App list, or CLAUDE template generation. The cleanest hooks are the phase gates, since they isolate behavior by concern: local repo, remote repo, app access, and Claude bootstrap.

Artifacts

Expected outputs are a git repository, GitHub remote, compliance topics, CODEOWNERS ownership mapping, three app connections, a generated CLAUDE guide, and a private or shared .claude/ directory depending on user preference.

Idempotency Strategy

The skill uses existence checks instead of optimistic writes. Existing git repos, remotes, CODEOWNERS, and CLAUDE files are treated as authoritative. This makes it safe to re-run after partial failures, especially when external setup completes asynchronously or manually.

Operational Risks

The most fragile segment is GitHub App connection, because it depends on token scope and org installation permissions that may sit outside normal repository admin rights. The skill explicitly surfaces this boundary and provides manual fallback URLs instead of masking access failures.

Internals Summary

Primary Contract

Turn an empty or partially initialized directory into a repo that is usable by both humans and automation, without clobbering prior work.

Best Modification Surface

Adjust the per-phase checkpoints and prompts first. Those boundaries keep changes localized and preserve safe retries.

Most Sensitive Boundary

GitHub App installation management sits outside ordinary repo CRUD and is the highest-friction part of the flow.

Internal takeaway: this skill is less a script than a resumable provisioning protocol. Its architecture is dominated by state detection, explicit confirmations, and cross-tool coordination rather than complex branching logic.
