Architecture Overview

Orchestration Layer

The skill acts as an orchestrator, calling plan_review_dispatch.py which spawns 3 CLI agents (Claude, Codex, Gemini) via subprocess.run with ThreadPoolExecutor. Each agent reviews 7 domains = 21 parallel sub-agents per round.

Review-Fix Loop

Up to N rounds (default 3). Each round: dispatch → classify findings → fix plan. Early termination when zero actionable findings. Final review-only round validates fixes.

Config Hierarchy

Global ~/.claude/code-review/config.json → Org .code-review/config.json → CLI args. max_rounds, fix_threshold, disabled_domains, timeout.

Finding Classification Pipeline

fix ACTION

Severity ≥ fix_threshold (default: medium) AND the issue genuinely exists in the plan. These get applied as edits in Phase 2c.

recurring ESCALATED

Same section + same domain flagged in a prior round that was supposedly fixed. Indicates incomplete or ineffective fix. Gets priority attention.

false_positive NOISE

Described problem doesn't exist in the plan or is already addressed elsewhere. Counted as noise, not issues.

noise NOISE

Subjective, stylistic, or single-agent finding contradicted by the other 2 agents. Filtered out of issue counts.

ignored BELOW THRESHOLD

Below fix_threshold (e.g., low severity when threshold is medium). Reported but not acted on.

high_confidence SIGNAL

Cross-reference boost: 2+ agents flagging the same section with the same concern. Applied as a modifier on top of classification.

PR Comment Architecture

Multi-Identity Posting

Each of the 3 GitHub Apps posts its own raw findings under its bot identity. The orchestrator summary posts under stark-claude[bot].

BotPostsAuth
stark-claude[bot] Claude's raw findings + orchestrator summary github_app.py --app stark-claude token
stark-codex[bot] Codex's raw findings github_app.py --app stark-codex token
stark-gemini[bot] Gemini's raw findings github_app.py --app stark-gemini token

If an agent returned 0 findings or failed, a short status comment is still posted under its identity. Posting failure for one agent does not block the others.

Failure Modes & Recovery

Total Dispatch Failure

Trigger: summary.succeeded == 0
Response: Run diagnostics (which claude/codex/gemini, single-agent probe). Skip remaining rounds + Phase 3. Generate dispatch-failure summary template. Do NOT treat as clean plan.

Low Coverage

Trigger: succeeded/total < 50%
Response: Warn "Low coverage — only N/M sub-agents succeeded. Results may be incomplete." Continue normally with available results.

Auth Failure

Trigger: github_app.py token fails when PR exists
Response: Warn "Could not authenticate, skipping PR posting." Skill continues — review still runs, just no PR comments.

Crash Mid-Round

Trigger: Process dies during Phase 2
Response: in-progress.json persisted per round enables manual recovery. Re-run will start fresh but history is preserved.

PR Comment Failure

Trigger: One bot's posting fails
Response: Warn and continue. Other bots still post. Terminal output always available as fallback.

Uncommitted Changes

Trigger: git diff shows changes without --force
Response: Abort with message. User must commit/stash or pass --force.

Extension Points

Adding a Domain

Create global/prompts/plan-review/{agent}/NN-domain.md for each agent (claude, codex, gemini). The dispatch script auto-discovers domains from filenames. Domain ID is derived from filename slug.

Disabling Domains

Set disabled_domains: ["domain-slug"] in repo-level .code-review/config.json. The dispatch script filters these out before spawning sub-agents.

Tuning fix_threshold

Config key plan_review.fix_threshold controls minimum severity for auto-fix. Values: low, medium (default), high. Higher = fewer auto-fixes, more left as unresolved.

Custom Agent CLI Flags

Edit scripts/plan_review_dispatch.py to modify CLI invocation per agent. Each agent's flags are in separate code paths. Smoke tests in test_plan_review_dispatch.py::TestCLIFlagsSmoke verify flag compatibility.

Repo-Level Prompt Overrides

Place overrides in {repo}/.code-review/plan-prompts/{agent}/{domain}.md. These take precedence over global prompts for that repo only.

Prompt Improvement Loop

Phase 4i generates a Prompt Improvement Assessment with recommended changes (global vs repo level). Use /stark-review-improvement to act on these recommendations.
