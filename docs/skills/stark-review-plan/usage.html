stark-review-plan

Multi-agent design document review for plans and specs, with optional autonomous fix rounds.
USAGE VIEW

What it does: You point the skill at a markdown plan or spec, and it runs 21 reviewers in parallel across 3 LLMs × 7 domains. It can classify findings, patch the plan for several rounds, run one final review-only pass, then produce a terminal summary, a .review.md file, PR comments, and history artifacts.

Common Workflows

Basic review with fixes

Run /stark-review-plan docs/specs/auth.md when you want the skill to review the plan, patch the document through up to three rounds, and then give you a final unresolved-issues summary.

Preview findings without edits

Run /stark-review-plan docs/specs/auth.md --dry-run when you want signal only. This skips file edits, skips PR posting, and does not write the sidecar review file.

Review an in-progress draft

Run /stark-review-plan docs/specs/auth.md --force if the plan is not committed yet but you still want the workflow to proceed on the current working copy.

Allow more repair cycles

Run /stark-review-plan docs/specs/auth.md --rounds 5 when the spec is broad or immature and likely needs more than the default three fix rounds before the final pass.

What the Skill Produces

Terminal summary

Always prints a consolidated markdown summary. In the healthy path this includes counts, findings table, fixed and unresolved items, noise analysis, diff summary, prompt-assessment guidance, metrics, and improvement flags.

Sidecar review file

When not in dry-run mode, writes a peer file such as design.review.md next to the reviewed plan so the review record lives beside the source document.

PR comments by bot identity

If a PR is detected and auth succeeds, each agent family posts its raw findings under its own bot account, then the orchestrator posts the classified summary.

History for recovery and analysis

Creates review history under the plan-review history directory, including rounds.json, summary.md, and temporary in-progress state for crash recovery.

Observability You Will See

Task-based progress

Five top-level tasks are created up front for setup, review-fix loop, final review, summary, and output. Additional round-specific child tasks appear dynamically as each dispatch and classify+fix segment starts.

Timestamped logs

Every phase transition and important round event prints with a wall-clock timestamp, so long runs remain traceable without guessing where the workflow is stuck.

Five-minute checkpoints

For long-running reviews, the skill must emit elapsed-time checkpoints with current phase, round, and partial sub-agent completion numbers.

Metrics and flags

The end of the run includes phase timing, issue and noise counts, agent success numbers, rounds used, and automated warnings such as bottlenecks or high agent failure rates.

Troubleshooting

“Usage: /stark-review-plan <path>”

The path argument is required. If you only passed a partial filename, the skill may search for close matches under docs/ and ask you to pick one.

Dirty plan blocked the run

The skill aborts when the plan file has uncommitted changes unless you explicitly pass --force. This protects against accidental edits to a draft you did not mean to mutate.

Zero successful sub-agents

This is handled as a dispatch failure, not as “no issues found.” Expect CLI availability checks, a single-agent probe, and a failure summary that points at auth, installation, or network problems.

Low coverage warning

If fewer than half of the sub-agents succeed, the review still continues, but you should treat the output as partial and investigate the failed agent family before trusting a clean result.
