Internal Architecture

generate_skill_docs.py

Main orchestrator script in scripts/. Handles CLI parsing (--check, --skill, --force), discovers SKILL.md files under skill/stark-*/, computes staleness by comparing source hash to generated output, and dispatches LLM generation calls per skill × audience.

LLM Tournament Engine

For each skill × audience pair, 3 LLMs (Claude, Codex, Gemini) independently generate HTML visualization, Mermaid diagram, doc-content JSON, and alt text. A tournament evaluator scores each submission and selects the winner. Scores are logged to docs/skills/_audit/scores.jsonl.

Change Detection

The --check flag walks all skill/stark-*/SKILL.md paths and compares content hashes (or mtimes) against the last-generated docs in docs/skills/<name>/. Returns exit code 0 if everything is fresh, exit code 1 with a list of stale skill names on stdout.

Output Structure

Each skill gets a directory under docs/skills/<name>/ containing two audience sets: usage.* (end-user) and internals.* (contributor). Each set has 5 files: .html (standalone viz), .md (markdown doc), .json (structured content), .mermaid (diagram source), .png (screenshot).

Manifest & Index

docs/skills/_manifest.json tracks all generated skills with metadata (last generated timestamp, winner LLM, scores). docs/skills/index.md and docs/skills/README.md are regenerated as human-readable indexes linking to each skill's docs.

Playwright Screenshots

PNG files are generated by rendering the HTML output in a headless Chromium browser via Playwright. If Playwright is not installed, the script continues without PNGs and emits a warning. This is the only optional external dependency.

Extension Points

Adding a New Audience

The system currently generates two audiences: usage and internals. To add a third (e.g., ops), add a new audience prompt template to the generation script and update the output file naming pattern. The tournament runs independently per audience, so no scoring changes are needed.

Adding/Swapping LLMs

Each LLM is a contestant in the tournament. To add a fourth LLM or replace one, update the contestant list in generate_skill_docs.py. Each contestant needs a callable that accepts (skill_content, audience, css) and returns (html, mermaid, json, alt_text).

Custom CSS Design System

The CSS is passed inline to each LLM as part of the prompt. To change visual style, edit the shared design-system CSS block. LLMs are instructed they may extend but not replace it, so additions are safe. Breaking changes to class names require updating all existing generated HTML files.

Tournament Scoring

The evaluator scores candidates on multiple dimensions (accuracy, completeness, visual quality, design-system compliance). Scores are appended to scores.jsonl for trend analysis. To change scoring criteria, modify the evaluation prompt in the tournament engine.

Failure Modes & Recovery
Failure Detection Recovery Impact
LLM call fails Exception or timeout from API call Report which LLM failed, continue tournament with remaining contestants Reduced competition quality; minimum 1 LLM needed
Playwright missing ImportError or browser launch failure Skip PNG generation, emit warning, write all other outputs No .png files; HTML/MD/JSON/Mermaid still generated
No changes detected --check returns exit 0 Report "all skill docs are up to date" and exit No-op; safe to run repeatedly
Malformed LLM output Missing required sections (HTML/Mermaid/JSON/alt text) Disqualify that contestant from tournament; log parse error Contestant excluded; others continue
All LLMs fail Zero valid contestants for a skill Skip that skill, report error, continue with remaining skills Stale docs remain for that skill
