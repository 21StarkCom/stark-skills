Release Workflow
Input: Must be on main branch with clean working tree
1. Pre-flight Check
git checkout main, pull --rebase, verify clean tree
ABORT if dirty tree or not on main
Source: git tags (semver). Baseline 0.1.0 if no tags
2. Determine Current Version
git tag --sort=-v:refname | head -1
Reads: CHANGELOG.md [Unreleased] section
3. Review Unreleased Changes
Extract and present Fixed / Added / Changed entries
ABORT if no CHANGELOG or empty [Unreleased]
Auto-detect: Fixed→patch, Added→minor, Breaking→major
4. Determine Bump Type
Use argument if given, otherwise auto-select from changes
Or explicit: /stark-release patch|minor|major
File: src/infra_pulse/__init__.py
5. Bump Runtime Version
Update __version__ = "X.Y.Z" (runtime source of truth)
Move: [Unreleased] entries → [vX.Y.Z] section
6. Update CHANGELOG & Commit
git add + commit "release: vX.Y.Z"
Important: Commit before tag so tag points to release commit
7. Create Annotated Tag
git tag -a vX.Y.Z -m "Release vX.Y.Z" with change list
ERROR if tag already exists
Pushes: main branch + tag
8. Push to Remote
git push origin main && git push origin vX.Y.Z
Recovery: pull --rebase and retry on failure
Auth: User's PAT via gh (unset GH_TOKEN)
9. Create GitHub Release
gh release create vX.Y.Z with CHANGELOG notes
Shows as: created by user, not bot
10. Release Summary
Version, tag, previous, change counts, GH Release URL, commit hash

Common Patterns

Quick Patch Release

After fixing a bug and updating CHANGELOG:
/stark-release patch
Bumps patch version, tags, creates GitHub Release — all automated.

Feature Release (Auto-detect)

After adding features with ### Added entries in CHANGELOG:
/stark-release
Auto-detects minor bump from Added entries. No argument needed.

Major Version Bump

For breaking changes or major milestones:
/stark-release major
Explicitly specify major to bump e.g. 0.3.0 → 1.0.0.

Outputs & Artifacts

Git Tag

Annotated tag vX.Y.Z with release message listing all changes from CHANGELOG.

Release Commit

Commit on main updating CHANGELOG.md and __version__. Tag points to this commit.

GitHub Release

Published release with CHANGELOG content as notes. Created under user's identity (not bot).

Failure Modes & Recovery
FailureWhat HappensRecovery
Not on main Skill aborts immediately git checkout main
Dirty tree Skill aborts immediately Stash or commit changes first
No CHANGELOG Skill aborts — nothing to release Create CHANGELOG.md with [Unreleased] entries
Empty [Unreleased] Skill aborts — nothing to release Add entries under [Unreleased] first
Tag exists Error — version already taken Skill suggests the next available version
Push fails Tag and commit created locally git pull --rebase origin main, retry push
gh auth fails GitHub Release not created Verify gh auth status — PAT must be active
Important Rules

Don't Bump pyproject.toml

Git tags are the version source of truth. The pyproject.toml version is static. Only __init__.py gets updated.

Commit Before Tag

Always commit the CHANGELOG + version bump first, then create the tag — so the tag points to the release commit.

User Identity, Not Bot

Releases use the user's PAT via native gh auth. Always unset GH_TOKEN before release operations.
