Version Resolution Logic

Version Source of Truth

Git tags (semver format vX.Y.Z) are the canonical version. The pyproject.toml static version = "0.1.0" is never modified. Runtime version lives in src/infra_pulse/__init__.py as __version__ and MUST match the tag.

Bump Auto-Selection

When no explicit argument is provided, the skill inspects CHANGELOG categories:
Only ### Fixed → patch
Has ### Added → minor
Has ### Changed with breaking changes → major
No user confirmation required — proceeds automatically.

Tag Parsing

git tag --sort=-v:refname | head -1 retrieves the most recent semver tag. Parsed into major, minor, patch integers. If no tags exist, falls back to baseline 0.1.0. Tag collision (version already taken) triggers an error suggesting the next available version.

Authentication Model

User's PAT (Exclusive)

All release operations — commits, tags, pushes, and GitHub Releases — use the user's personal access token via native gh auth. GH_TOKEN is explicitly unset at the start to prevent bot tokens from being used. This ensures releases show the user as the actor, not a bot.

No Bot Tokens

Unlike /stark-review which uses stark-claude[bot] for review comments, the release skill never touches bot credentials. The unset GH_TOKEN at pre-flight is a critical guard — without it, a stale bot token from a prior review could cause releases to appear as bot-authored.

Extension Points

Version File Location

Currently hardcoded to src/infra_pulse/__init__.py. To adapt for other projects: change the file path and the regex pattern used to find/replace __version__. Could be generalized to read from a config field specifying the version file path.

Dynamic Versioning

The skill notes that pyproject.toml is static. To enable dynamic versioning, add setuptools-scm which reads version from git tags at build time. This would eliminate the need for Step 5 entirely — the tag itself becomes the only version artifact.

CHANGELOG Format

Expects Keep a Changelog format with ## [Unreleased] heading and ### Fixed/Added/Changed subsections. The bump auto-selection logic is tightly coupled to these category names. Adding new categories (e.g., ### Deprecated, ### Removed) requires updating the bump heuristic.

Observability Protocol

The skill follows the Skill Observability Protocol at ~/.claude/code-review/standards/observability.md. Custom metrics include: version diff (prev → new), bump type, CHANGELOG entry counts by category, tag/release creation booleans, and push duration. These can be extended or piped to external dashboards.

Post-Release Hooks

Currently ends at summary output. Natural extension points: trigger CI/CD deploy pipelines, notify Slack channels, update version badges, run smoke tests against the tagged release, or create follow-up issues for the next cycle.

Multi-Package Support

The skill assumes a single-package repo with one CHANGELOG and one version file. For monorepos, each package would need its own CHANGELOG parsing, independent version bumps, and scoped tags (e.g., pkg-a/v1.2.3).

Critical Ordering Constraints

Commit Before Tag

The tag must point to the commit that contains the updated CHANGELOG and __version__. Creating the tag first would point it at the wrong commit.

Unset GH_TOKEN First

Must happen at pre-flight before any git/gh operations. A stale bot token from /stark-review could silently cause releases to appear bot-authored.

Push Both Separately

Branch push and tag push are separate commands. Both must succeed. Tag push after branch push ensures remote has the commit the tag references.
