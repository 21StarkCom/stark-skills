stark-session

Session lifecycle orchestration for contributors: context recovery at start, release hygiene at end
INTERNALS • ARCHITECTURE / DATA FLOW / EXTENSION POINTS
This skill is a two-mode controller. Start mode pulls working context into a concise briefing; End mode closes the loop with verification, merge, documentation, project metadata updates, and push behavior guarded by user confirmations.

Trigger Surface

Invoked by conversational prompts like “session start”, “catch me up”, “session end”, or explicitly via /stark-session [start|end]. Default mode is start.

Primary State Inputs

Hierarchical config, CLAUDE.md ancestry, memory files, git state, PR metadata, optional GitHub project board state, and local skill discovery.

Primary Artifacts

Start produces a briefing. End produces test/build results, optional PR merges, a docs commit, project field updates, push state, and a final completion summary.

Control Plane

Start Mode

Context acquisition pipeline with silent reads, non-fatal health checks, and a short briefing optimized for re-entry into active work.

End Mode

Exit pipeline that validates work, reconciles PR state, captures docs/devlog changes, updates project fields, and handles branch push policy.

End Mode Flow
The end path is a guarded shutdown sequence: verify first, ask on risky transitions, and never silently discard failure information.
Inputs

Session config, top-level test fallback, local repo state.
Phase 1: Run Tests and Build
Execute configured test/build commands and record pass/fail durations
Decision Gate

Failures trigger “Proceed anyway?” rather than an automatic stop.
External

Uses the user's gh auth and PAT, never the GitHub App bot.
Phase 2: Merge Open PRs
Discover branch PRs and offer merge with configured squash/merge/rebase strategy
Recovery

Auth failure or merge failure downgrades to warning and user choice.
Config-Driven

doc_paths, optional devlog_path.
Phase 3: Commit Docs
Stage docs, optionally create dated devlog entry, ask for a one-line summary, commit only if staged diff exists
Guardrail

Never creates an empty commit.
Optional External

Reads .github/project-config.json if present.
Phase 4: Project Field Updates
Infer touched issues from commits, draft docs state when appropriate, preserve manual reviewed/complete values, validate artifact links for higher-risk items
Guardrail

Warnings never block session end and manual state is not overwritten.
Branch Policy

Behavior depends on upstream tracking and whether branch is main/master.
Phase 5: Push Decision
Push tracked branches automatically; ask before pushing untracked non-main branches
Recovery

Push failures are reported explicitly and handed back to the user.
Output

Compact end-of-session operational ledger.
Phase 6: Summary
Tests, PR merges, docs commit count, project updates, push destination, session complete
Observability

Durations and counts align with the observability protocol.

Extension Points

Health Check Expansion

Add repository-specific smoke tests, schema checks, or environment probes by extending session.health_checks. They remain advisory and naturally fold into the briefing.

PR Merge Policy

Change session.pr_merge_strategy to alter merge semantics without rewriting the workflow. The decision stays localized to Phase 2 of end mode.

Documentation Automation

Refine doc_paths and devlog_path to control which artifacts become part of the end-of-session commit and how session summaries are preserved.

Project Board Integration

Dropping in .github/project-config.json activates board reads on start and field reconciliation on end. The main skill stays operational when this file is absent.

Briefing Surface Area

Adding local skills to .claude/skills or user skills to ~/.claude/skills automatically broadens the suggested tool surface without any code changes in this skill.

Failure Policy

The skill is deliberately fail-soft. Any future modification should preserve the contract that test, auth, project API, and push problems produce warnings and decision prompts rather than silent aborts.

Failure and Recovery Model
Failure Where It Appears Recovery Behavior
No CLAUDE.md Start / context Note it in the briefing and suggest onboarding; continue normally
No memory files Start / context Report no prior context; not an error
gh auth failure Start PR read, End PR merge Skip PR operations with a warning
Not a git repo Start / git state Skip git and PR sections
Health check failure Start / health Include stderr in briefing and continue
Test or build failure End / verification Ask whether to proceed anyway
PR merge failure End / PRs Offer to skip the failed PR and continue
Push failure End / push Report error and hand control to the user
Project API/config missing Start and End / project integration Skip or log warnings, never block completion
Internal Design Notes

Data Flow Shape

The skill is a fan-in/fan-out controller. It fans in from local files, git, GitHub CLI, project metadata, and skill directories; then fans out to either a human briefing or operational side effects like merge, commit, project updates, and push.

Files Git gh Project API

Operator Safety Model

Potentially destructive or irreversible steps are wrapped in prompts: proceeding after failed verification, merging PRs, skipping failed merges, and pushing branches without upstream tracking. That is the core user-trust mechanism.
