stark-pr-flow

Internal PR orchestration for GetEvinced repos: user-auth mutations, bot-auth review, and gated admin squash-merge
INTERNALS • ARCHITECTURE / DATA FLOW / EXTENSION POINTS

This skill wraps the full contributor-facing PR lifecycle into a single controlled sequence: validate branch state, push, inspect diff, generate PR metadata, create the PR with the user's native gh auth, post a self-review through stark-claude[bot], stop for explicit approval, then optionally merge with --squash --admin and clean up branch state.

The core architectural constraint is the auth boundary: do not export GH_TOKEN for PR creation or merge. Only the self-review post and project advisory lookup are allowed to cross into bot-token execution, so the visible actor on review comments becomes stark-claude[bot] without hijacking the user's normal GitHub CLI identity.

Extension Points

PR Metadata Generation

Title and body synthesis is the cleanest customization seam. You can change the title policy, section ordering, checklist template, or scope language without touching auth or merge semantics.

Review Heuristics

The self-review body can be expanded with repo-specific checks, inline comment policies, severity tagging, or required test evidence. The posting channel stays the same: stark_claude.pr_review(...).

Advisory Project Checks

Documentation State is already modeled as a non-blocking lookup. Additional project fields or issue-derived validations can be inserted here as long as they only warn, not mutate or gate merge implicitly.

User Argument Surface

The current interface only recognizes title override text and the literal draft keyword. New flags should be added carefully so they do not blur the current safe default of non-draft PR creation and explicit merge approval.

Merge Policy

Today the merge contract is fixed: --squash --admin. If that policy changes, update the final phase and the stop-summary wording together so operator expectations stay aligned with actual mutation behavior.

Failure Reporting

Recovery paths are explicit but mostly procedural. A natural improvement is richer status surfacing: detect current failure mode, print exact next command, and record that checkpoint in observability metrics.

Failure Topology

Hard Stops

Main branch usage and dirty worktrees terminate execution immediately. These failures protect branch hygiene and prevent accidental PRs from unstable local state.

Recoverable Branch Issues

Diverged pushes and merge conflicts route through rebase-on-main, conflict resolution, and retry. The workflow assumes contributors can repair branch state and resume.

Auth/Service Faults

gh auth status diagnoses user PAT failures; the scripts venv and GitHub App token path diagnose bot-review failures. The two auth lanes stay independently debuggable.

Internal Notes

Data Flow Summary

Local git state produces branch name, commit delta, and diff. Diff analysis yields PR title/body artifacts. PR creation yields PR number and URL. Those artifacts are then passed into bot review posting. After explicit approval, merge output drives local and remote branch deletion. Each stage consumes the previous stage's artifact rather than recomputing global state.

Key Invariants

  • PRs are not draft unless explicitly requested.
  • Self-review is mandatory before merge.
  • Merge never occurs before an explicit user signal.
  • All merge operations use admin squash strategy.
  • System Python is intentionally avoided for bot scripts.
