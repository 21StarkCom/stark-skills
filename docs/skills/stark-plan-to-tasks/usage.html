stark-plan-to-tasks

Decompose specs into phased GitHub issues and extract architectural knowledge.
Audience: Usage

Invocation

Invoke via natural language or explicitly via CLI:

/stark-plan-to-tasks <path-to-spec> [options]

Natural language: "break down this plan", "create issues from spec", "plan to tasks"

Options

  • --dry-run : Run all 3 LLM passes, validate, preview the output to a temporary markdown file, but stop before modifying any repo files or creating issues.
  • --cleanup <slug> : Find and close obsolete issues labeled with a specific plan slug.

The User Journey (End-to-End Workflow)

User provides a `.md` plan file. System authenticates.
1. Setup & Ingestion
Reads plan, detects target GitHub repository
2. Plan Quality Gate (LLM Pass 1)
Checks for missing decisions, vague paths, completeness
Checklist Failure: Skill pauses and asks user to clarify missing details in the plan file.
Plan broken down into logical phases and scoped tasks.
3. Decomposition (LLM Pass 2)
Generates JSON phases, tasks, story points, constraints
4. Validation (LLM Pass 3)
Adversarial agent checks dependencies, coverage, sizing
Ensures no orphans, no circular dependencies, proper sizing.
Output Artifacts:

- Labels created

- Phase Issues created

- Task Issues created
5. GitHub Issue Creation
Auth as user, create issues, link dependencies, add to GH Projects
6. Knowledge Extraction
Extracts schemas, API contracts, ADRs into docs/ directory
Cleanup: Deletes the original plan.md file and performs a local git commit.

What Gets Generated? (Outputs)

GitHub Issues & Labels

Creates Phase tracking issues and deeply detailed Task issues. Auto-provisions required labels:

  • Sizing: sp:1 to sp:13 (Fibonacci)
  • Risk: risk:low, risk:med, risk:high
  • Confidence: confidence:low to high
  • Traceability: plan:slug

Uses native GH Types (Bug, Feature, Task).

GitHub Project Integration

Automatically maps tasks to fields (if .github/project-config.json is present):

  • Status: Backlog / Ready for Agent
  • Phase: Matches decomposition
  • AI Suitability: Autonomous / Assisted / Human-led
  • Priority: Derived from risk

Living Documentation

Deletes the transient plan file and persists the architectural decisions locally. Performs a local commit with:

  • docs/decisions.md (Tracking record)
  • docs/adr/* (Architectural decisions)
  • docs/data-model.md (Schemas)
  • docs/api.md (API Contracts)

Issue Classifications

Classification Values Description
GitHub Issue Type Feature, Task, Bug Feature = new capability. Task = refactoring/infra/docs. Bug = fixing explicit defects.
AI Suitability Autonomous, Assisted, Human-led Autonomous = AI can do it 100%. Assisted = Needs guidance. Human-led = Needs PM/UX input.
Story Points 1, 2, 3, 5, 8, 13 Fibonacci scale based on complexity and unknowns.
Risk & Confidence Low, Medium, High Evaluates technical risk and the LLM's confidence that the task is fully specified.
