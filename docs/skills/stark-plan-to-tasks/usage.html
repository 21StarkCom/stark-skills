stark-plan-to-tasks

Turn a markdown spec into phased GitHub issues, validate the breakdown, extract durable knowledge, then remove the plan.
USAGE VIEW · complex skill · 3 LLM passes + GitHub workflow
What users invoke

This skill is for the moment when a design doc is done and you want execution-ready work items. It reads one `.md` plan, blocks on missing detail, decomposes it into phases + task issues, labels and types each issue, optionally previews everything in dry-run mode, then enriches project docs and deletes the source plan.

What You Must Provide

Input File

The first argument is required, must exist, must be non-empty, and must end in .md. The skill reads the full plan and treats it as the source of truth.

GitHub Access

gh must be installed. The stark-claude GitHub App must be installed on the target repo with write access for checks, labels, and project mutations.

Validation Agent

Configured validation CLIs such as codex must exist in PATH. Missing agents fail early before decomposition continues.

Plan Quality

The plan must already answer concrete implementation questions. This skill will not invent architecture or fill in vague sections.

Issue Payload Shape

Per Task Fields

  • Identity: stable task_id, title, phase
  • Context: what, why, where, how
  • Done signal: up to 5 acceptance criteria
  • Risk model: story points, risk, confidence
  • Execution hint: AI suitability + review hints
  • Links: dependency task IDs, then patched GitHub issue numbers

GitHub Issue Outcome

  • Native issue type: Task, Bug, or Feature
  • Labels: story points, risk, confidence, traceability, metadata
  • Phase tracking issue created first for checklist anchoring
  • Run manifest stored in the breakdown temp file for resumability
  • Optional GitHub Project fields populated after creation
Common User Workflows

1. Preview First

Run with --dry-run when the plan is probably good but you want to inspect sizing, labels, and issue wording before any GitHub mutation.

2. Rerun an Updated Plan

If issues already exist for the derived plan slug, the skill stops and asks whether to skip, update existing issues, or create a fresh parallel set.

3. Clean Up Older Runs

Use --cleanup <slug> to find traceable issues by plan:{slug} and close them with a standard cleanup comment.

Guardrails That Matter To Users

No Silent Progress on Bad Plans

Weak plans stop at the quality gate after at most three user revision rounds. The skill refuses to invent missing implementation detail.

No Issue Creation on Failed Validation

If validation cannot converge after two fix iterations, issue creation is blocked. Remaining problems are surfaced instead of producing shaky tickets.

Creation Is Resume-Friendly

Partial issue creation is tracked in a manifest so a rerun can skip already-created task issues instead of duplicating everything blindly.

Plan Deletion Happens Late

The plan is only removed after successful issue creation and doc enrichment. If the commit fails, the plan is intentionally not deleted.

Outputs You See

In GitHub

Phase tracking issues, task issues, labels, dependency links, and possibly project field updates if .github/project-config.json exists.

In Your Repo

Appended or newly created docs under docs/, a decision record entry, and a local commit that adds only the touched documentation files plus the removed plan.

In Temp and Logs

Breakdown JSON, validation envelope, optional preview markdown, and a JSONL observability record with durations, counts, and distributions.

Fast Mental Model

The skill behaves like a controlled funnel

It narrows a single approved plan into a validated execution graph. Every stage either strengthens confidence or aborts with a concrete reason. For end users, the key choice is simple: preview with --dry-run if you are unsure, or run normally when the plan is already implementation-grade.

Best for approved specs Not for brainstorming Best when repo + gh auth are already healthy
Native GitHub Issue Types are used for Task, Bug, and Feature. The skill explicitly avoids creating type labels like type:task.
