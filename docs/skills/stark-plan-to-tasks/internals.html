stark-plan-to-tasks

Spec-to-issues orchestration with validation, rerun safety, doc extraction, and GitHub project integration
INTERNALS · architecture, control flow, artifacts, extension points

This skill is effectively a stateful workflow engine wrapped in a markdown skill: it validates a spec, decomposes it into phased issue payloads, adversarially re-checks that breakdown with external agents, then mutates GitHub and the local repo only after integrity gates pass.

Core Artifacts & State Transitions

Plan File as Mutable Input

The spec markdown is read repeatedly, not cached blindly. Pass 1 can force up to three user-edit cycles, and Pass 4 re-hashes the file before validation dispatch so decomposition cannot drift from the current plan.

Breakdown JSON as Durable Work Packet

A temp JSON file with schema_version, plan_hash, phases, tasks, and later a _manifest. This is the handoff boundary between decomposition, validation, issue creation, and rerun recovery.

Validation Envelope

The skill packages raw plan markdown, breakdown JSON, and the plan hash into a single envelope so the external validation agent can assess completeness, overlap, dependency integrity, and sizing without hidden context.

Run Manifest Enables Idempotency

Every created issue appends {task_id, issue_number, phase_id}. On rerun, the skill cross-references existing issues and the manifest so partial runs can resume instead of duplicating work.

Docs Tree as Routing Map

The local docs/ tree is not just documentation output; it acts as a content router. Architectural decisions, data models, APIs, constraints, and glossary terms each pick append destinations or create fallback files.

Plan Deletion is Late-Bound

Deletion is intentionally deferred until issues exist and enrichment succeeds. If commit fails, the plan stays, avoiding the worst failure mode: losing source intent without durable downstream artifacts.

Config, Defaults, and Extension Points

Validation Agent Stack

Reads validation_agents from global → org → repo config, defaulting to ["codex"]. Each configured binary must exist in PATH before any LLM pass begins.

default: ["codex"]

Project Integration Toggle

Presence of .github/project-config.json opt-ins the entire GitHub Project phase. Without it, project mutations are skipped cleanly and the core issue workflow remains unchanged.

extension point: repo-level opt-in

Docs Routing Heuristics

Knowledge extraction checks well-known doc locations first, then creates fallback files such as docs/api.md or sequential ADR entries when no natural home exists.

append-only, never merge/replace

Native GitHub Issue Type Mapping

feature, task, and bug map to org-level GitHub Issue Types via --field type=.... The skill explicitly forbids creating synthetic type:* labels, which keeps it aligned with the org's canonical issue taxonomy.

Prompt-Embedded Validator Contract

The separate-agent validator is forced into a strict JSON output schema and a fixed checklist. That makes agent outputs parseable, retryable, and comparable across providers rather than conversational and brittle.

Critical Control Logic

Why This Skill Is Operationally Heavy

It mixes three risk domains that are usually isolated: LLM generation, GitHub mutation, and local repo mutation. The design compensates with repeated integrity checks, explicit human checkpoints, temp-file contracts, and late execution of destructive actions.

  • Repo detection is repeated and compared against the current checkout.
  • Pass boundaries are explicit and backed by disk artifacts, not ephemeral memory.
  • Mutation order is carefully staged: issue bodies first use placeholders, then a patch pass resolves dependency numbers.
  • Project integration is additive and non-blocking.
  • Knowledge extraction is downstream from issue creation, never upstream.

Failure Semantics

The workflow is intentionally asymmetric: early phases fail hard, late phases fail soft where possible.

  • Setup/auth/repo access: hard stop.
  • Quality gate and validation non-convergence: hard stop.
  • Project field mutation: warn and continue.
  • Commit failure after enrichment: preserve plan and leave changes unstaged.

Dirty tree or commit hook failure during enrichment Phase 6 Warn, preserve plan, leave enrichment changes for manual review.
What Contributors Would Change

Change Breakdown Granularity

Adjust the task schema constraints and sizing guardrails if you want larger or smaller issues. The decomposition pass and validation pass must stay in sync or the validator will reject the new task shape.

Add New Validation Agents

Extend validation_agents and the normalization logic in the validator wrapper. The rest of the skill already supports multi-agent union semantics for findings.

Reroute Knowledge Extraction

Modify the docs routing matrix and ADR fallback behavior if your repo uses a different documentation topology. The append-only rule is the safety invariant worth preserving.

Evolve Project Fields

Project integration is the cleanest extension seam: field names and value derivation live after issue creation, and failures there do not compromise the main workflow.
