stark-pr-status

\n
\n Internal PR analytics skill that joins GitHub review metadata with stark-review history,\n then projects a terminal-first dashboard for review rounds, severity mix, signal-vs-noise,\n participants, merge timing, and follow-up actions.\n

Primary Job

\n

\n Explain what happened on a PR, not just its current state. The skill reconstructs review\n rounds, severity distribution, outcome quality, participant behavior, and merge timing.\n

\n
\n review rounds\n severity mix\n time-to-merge\n
\n
\n\n
\n

Data Blend

\n

\n GitHub provides canonical PR, review, and participant events. stark-review history adds\n findings, noise classification, and historical review signal that GitHub alone does not expose.\n

\n
\n github-only\n history-only\n both\n
\n
\n\n
\n

Presenter Contract

\n

\n The Python script owns formatting. Single-PR mode emits a rich dashboard; all-PR mode emits a\n compact table; --json exposes a machine-readable variant for downstream consumers.\n

\n
terminal-first skill
\n
\n
\n

Runtime Contract

\n
\n
\n

Invocation Surface

\n

\n Entry point is pr_status.py with positional PR selection or --all.\n Repo detection defaults to the current git checkout unless --repo org/name is supplied.\n

\n
\n
\n

Mode Switches

\n

\n --state open|closed|merged|all constrains the PR set, --limit N caps\n fleet summaries, and --json bypasses human-oriented rendering for structured output.\n

\n
\n
\n

Post-Analysis Heuristics

\n

\n After rendering, the skill inspects stale PRs, high-noise patterns, and merge-ready conditions\n to suggest adjacent skills such as /stark-review, /stark-review-improvement,\n and /stark-pr-flow.\n

\n

Failure Modes And Recovery

\n
\n
\n

No Repo Detected

\n

\n Exit 1. Triggered when the current directory is not a git repo and no --repo override exists.\n Recovery is explicit: run from a repo or pass --repo org/name.\n

\n
\n
\n

PR Not Found

\n

\n Exit 2. Single-PR lookups fail fast when the requested PR number does not exist in the selected repo.\n The UI should preserve the repo context in the error text.\n

\n
\n
\n

GitHub API Failure

\n

\n Exit 3. Covers auth failures, upstream API issues, and rate limiting. GitHub state is canonical, so the\n skill cannot safely reconstruct the dashboard without it.\n

\n
\n
\n

No History For PR

\n

\n Non-fatal. The skill still shows GitHub-derived review activity and clearly notes that stark-review history\n is unavailable, preventing silent gaps in analytics quality.\n

Observability And Extension Points

\n
\n
\n

Observability Protocol

\n

\n Emit the shared Skill Observability Protocol and add skill-specific counters for PRs analyzed, GitHub API calls,\n history records loaded, and whether the result was sourced from GitHub, history, or both.\n

\n
\n
\n

Analytics Extension Point

\n

\n New metrics should land in the lifecycle aggregator first. That keeps single-PR dashboards, fleet summaries,\n and JSON output aligned instead of duplicating business logic across presenters.\n

\n
\n
\n

Presenter Extension Point

\n

\n If you need richer automation, extend --json rather than scraping terminal output.\n The terminal contract remains stable because the script already owns preformatted display text.\n

\n
\n
\n

Workflow Extension Point

\n

\n Heuristic suggestions are intentionally isolated. Add org-specific rules for stale reviews, reviewer load,\n or CI gating without touching collection or normalization logic.\n
