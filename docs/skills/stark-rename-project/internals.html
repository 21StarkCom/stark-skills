Primary execution model

The skill behaves like a single-run state machine with resume detection. Inputs are parsed once, then every later phase assumes normalized OLD_NAME, NEW_NAME, REPO_ROOT, PARENT, HOST, and ORG. Mutation order is deliberate: remote rename first, local directory rename second, reference rewrites third, symlink reinstall last.

State domains touched

There are five mutable domains: GitHub repository metadata, local git remotes, local directory path, tracked text files in the renamed repo and siblings, and installed symlink destinations. Verification re-reads all five domains to catch drift or partial completion.

Critical invariants

Literal-safe replacements: Perl \\Q...\\E avoids regex surprises in repo names containing dots or hyphens.
Idempotent resume logic: remote/local name combinations determine restart phase without replaying already-finished mutations.
Safe sibling updates: only sibling repos under the same parent, same host, same org, and clean worktrees are eligible.
Symlink ordering: old symlinks are removed before any install script edits to avoid uninstall targeting the wrong path.

Config Hardcoded anchors

SCRIPTS=~/.claude/code-review/scripts and PYTHON=$SCRIPTS/.venv/bin/python3 provide the GitHub App token path. This is an implementation seam if token sourcing changes.

Branching Dry-run behavior

Every mutation phase has an explicit dry-run gate. Internally that means phase logic is dual-purpose: execution and explain-only preview share the same sequencing.

Failures Dirty sibling repos

Sibling repos are soft failures. They are skipped and reported rather than aborting the entire rename, which keeps the core project migration progressing.

Observability Output contract

The summary is not cosmetic. It is the operator handoff artifact: changed files by repo, residual old-name references, workflow leftovers, and manual integrations to revisit.

Failure Surfaces

Pre-flight blockers

Invalid argument characters, missing names, remote repo mismatch, uncommitted changes, existing target directory, lack of admin permission, or a different GitHub repo already owning the target name all stop the run before destructive mutation.

Partial completion handling

The skill expects real-world interruption. Resume logic prevents double-renaming remote/local state, while repo ID comparison distinguishes “same repo renamed already” from “different repo collided with target name”.

Known manual boundaries

CI/CD workflows are scan-only, not auto-edited. Webhooks, Slack, Jira, GitHub App names, skill command names, and repos outside the parent directory remain outside the mutation boundary and are surfaced in the summary.

Extension Points

Parser / platform layer

Extend remote parsing to support more git URL shapes, additional hosts, or enterprise GitHub instances. The current implementation already avoids hardcoded host/org literals, so this is the cleanest portability seam.

Mutation engine

Add new deterministic replacement classes or richer exclusion rules. Keep ordering strict: path and URL rewrites before heuristic bare-name edits, and preserve literal-safe matching.

Verification / observability

Attach more post-checks, richer metrics, or structured machine-readable summaries. This skill already exposes a checkpointed execution model well-suited for timing, counts, and residual anomaly reporting.

Key Takeaway

Why this skill is more than “rename the repo”

Internally, it is a constrained migration framework. The important logic is not the rename itself; it is the choreography around idempotency, exact-match replacement safety, sibling repo hygiene, symlink lifecycle ordering, and operator-visible residual reporting. That is what makes the workflow reliable under interruption, partial execution, and mixed local/GitHub state.
