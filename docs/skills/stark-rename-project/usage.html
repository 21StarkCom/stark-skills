stark-rename-project

Rename a local repo and its GitHub repo together, propagate the change into sibling repos, and rebuild symlinks safely.
USAGE VIEW • complex skill • <old-name> <new-name> [--dry-run]
End-user Workflow

This skill is for controlled repo renames, not just a folder rename. It validates the current repo, renames the GitHub repository, rewrites local references, updates sibling repos under the same parent directory, removes stale symlinks, reinstalls fresh ones, and then verifies what still needs manual review.

At A Glance

Inputs

<old-name>, <new-name>, optional --dry-run. Names must be safe repo names with no spaces, separators, or shell metacharacters.

Main Outputs

Renamed GitHub repo, updated origin URLs, renamed local directory, rewritten text references, sibling repo commits, refreshed symlinks, and a final verification summary.

Manual Follow-Up

Review reported CI/CD workflow files and any external integrations such as webhooks, Slack, Jira, or anything keyed to the old repo name.

Quick Start

Standard rename

Invoke the skill with the current repo name and the desired new name. The skill renames GitHub first, then updates local state and related repos.

/stark-rename-project stark-review stark-skills

Preview first

Use --dry-run to see what would be renamed, which symlinks would be removed, and which repos or files would be touched before any change is executed.

/stark-rename-project stark-review stark-skills --dry-run

Resume a partial rename

If GitHub and local state disagree, the skill detects the state and resumes from the right phase instead of blindly restarting from the beginning.

Examples: GitHub already renamed but local dir still old, or local dir already renamed but remote still old.

Workflow
Entry

User provides old and new names plus optional preview flag.
1. Parse Inputs
Validate names, detect --dry-run, require both arguments
Rejects unsafe names, missing args, spaces, separators, or shell metacharacters.
Finds repo root, parent directory, install.sh, and uninstall support.
2. Establish Context
Repo root, parent dir, sibling repo zone, symlink strategy
Uses the current repo as the source of truth, not a nested subdirectory.
Decision

Checks remote URL and local directory names for resumable states.
3. Resume Detection
Continue from GitHub rename, local rename, or symlink cleanup if partially completed
Avoids duplicate work when a previous run stopped midway.
Confirms clean worktree, parses remote host/org/repo, verifies GitHub admin permission, checks name availability.
4. Pre-flight Validation
Safety checks before any rename happens
Stops early if the repo is dirty or the target name already belongs to a different GitHub repo.
Dry run branch

Reports planned GitHub rename instead of executing it.
5. Rename On GitHub
PATCH repo name, verify returned name, then rewrite fetch/push URLs
Uses GitHub redirect behavior, but warns that redirects break if the old repo name is later reused.
Handles case-only renames via a temporary directory name when needed.
6. Rename Local Directory
Move repo folder and switch working path to the new absolute location
All later paths must use the new absolute location.
Runs uninstall before file edits so old install targets can still be found.
7. Remove Old Symlinks
Use install.sh --uninstall or exact-target fallback cleanup
Deletes only symlinks whose resolved targets point to the old repo path.
Tracked text files only. Workflow files are scanned and reported, not auto-modified.
8. Rewrite References In Renamed Repo
Update absolute paths, org/repo refs, clone URLs, and limited bare-name matches
Protects skill IDs, invocation names, GitHub App names, workflow files, and historical filenames.
Decision

Each sibling repo is checked for same host/org and clean status before edits.
9. Update Sibling Repos
Modify matching repos under the same parent directory and auto-commit changed files
Dirty sibling repos are skipped and listed in the summary instead of being force-modified.
Runs from the new repo location after references have been updated.
10. Reinstall Symlinks
Execute install.sh to recreate symlinks for the renamed project
If install fails, the summary tells the user where to rerun it manually.
Checks remote connectivity, stale symlinks, residual references, and workflow files needing manual review.
11. Verify + Summarize
Changed files, skipped repos, residual refs, CI/CD review list, final cd target
This is the user-facing deliverable after the rename finishes.

What You Need Before Running It

Repository state

The current repo should have a clean git worktree, a valid origin remote, and ideally be launched from somewhere inside the repo so the skill can resolve the repo root.

GitHub access

The GitHub App token flow must work, and the app must have Administration:write on the repo. The skill explicitly checks admin permission before attempting the GitHub rename.

Symlink support

install.sh should exist and preferably support --uninstall. If it does not, the skill falls back to removing only symlinks that resolve to the old repo path.

Common Workflows

1. Safe preview before touching anything

Preview Run with --dry-run.
See GitHub rename target, local path rename, symlink removals, and sibling repo impact.
Then Re-run without --dry-run once the plan looks correct.

2. Standard full rename

Check Clean worktree and correct repo names.
Run Rename on GitHub first, then local folder and references.
Finish Inspect the final summary for skipped repos and manual CI/CD follow-up.

3. Resume after an interrupted run

Detect Remote and local names disagree.
Resume The skill jumps to the right phase instead of restarting.
Validate Use the summary to confirm the resumed path completed cleanly.

4. Case-only rename

Use case Example: stark-Skills to stark-skills.
Handling The local rename uses a temporary intermediate directory name.
Result Works safely on case-insensitive filesystems.
Outputs You Should Expect

Final summary

Lists every changed file grouped by repo, reports verification pass/fail checks, identifies residual old-name references that need manual review, and prints the final cd command for the new repo location.

Sibling repo commits

Each clean sibling repo that needs updates gets a focused commit: chore: update references from old-name to new-name. Dirty sibling repos are skipped instead of folded into accidental commits.

Manual review queue

Workflow files in .github/workflows/ are only scanned, never auto-edited. Those hits appear in the summary so you can decide whether CI/CD identifiers should change.

Stop Conditions And Common Failures
Condition What happens User action
Missing arguments or invalid names The skill exits before any change. Provide both names and keep them to safe repo characters only.
Uncommitted changes in the current repo The skill refuses to proceed. Commit or stash first, then rerun.
No GitHub admin permission GitHub rename phase stops. Grant Administration:write to the GitHub App.
Target repo name already exists on GitHub Rename is blocked to avoid collisions. Choose a different target name or resolve the existing repo first.
Dirty sibling repos Those repos are skipped, not modified. Clean them and rerun if you want references updated there too.
install.sh missing or failing Symlink recreation is incomplete. Fix the script and rerun it manually from the new repo path.
External Dependencies

GitHub API

The rename is not purely local. The skill calls GitHub to rename the repository and confirm permission, availability, and exact final name.

Sibling repos under the same parent

The skill discovers peer repos by scanning the parent directory and only updates those whose remotes point to the same host and org.

Symlink install locations

Cleanup and reinstall can touch symlinks under common install destinations such as ~/.claude and related code-review paths.

What This Skill Will Not Do
It does not rename skill invocation names, does not auto-edit CI/CD workflow files, does not update repos outside the parent directory, does not rename GitHub Apps, does not modify untracked or binary files, and does not fix external integrations like webhooks, Slack, or Jira automatically.
