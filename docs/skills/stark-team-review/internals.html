stark-review

Multi-agent PR review orchestrator with autonomous fix loop, classified signal tracking, and post-run observability

Control Plane

The skill is an orchestration layer around Git, GitHub, a Python review dispatcher, repo config, and a persistent history store. It never reviews directly; it coordinates review agents, adjudicates output, applies fixes, validates builds, and posts the classified summary.

Round Loop Internals

Dispatch

One orchestrator call fans out to 18 sub-reviews. Per-round child tasks track dispatch progress and completion counts.

Classification

Findings are validated against real code, deduped by region/domain, and split into issue versus noise buckets.

Repair

Only actionable findings are fixed. Build and test retries are bounded to three attempts per round.

Memory

Recurring findings, timing, and agent outcomes are persisted so later rounds and postmortems have structure.

The core architectural idea is not "use many agents"; it is "use many agents, then adjudicate them with local code truth, preserve round memory, and separate signal quality from issue count."

Data Products

Round JSON

Stores full per-round evidence: findings, classifications, agent result envelopes, durations, and timing slices for dispatch, classify/fix, and build/test. This is the canonical machine-readable trace.

Summary Markdown

The operator-facing artifact: issue counts vs noise counts, findings table, fixed items, recurring items, false-positive rationale, misalignment root causes, and prompt-improvement recommendations.

Prompt Assessment

Not a prompt mutator. It recommends where to tune agent/domain prompt files or repo config to reduce false positives and missed classes of bugs.

Observability Contract

Task Graph

Phases 1-5 must be represented as explicit tasks, with only one active at a time. Round-specific child tasks are created lazily as the loop advances.

Timestamped Logs

Every phase transition and round event prints a timestamped line, including dispatch counts, classification totals, build/test outcomes, and commit/push completion.

Improvement Flags

The final block must call out bottlenecks, high agent failure rate, empty actionable rounds, and excess build/test retries. Silence is explicit: "No improvement opportunities detected."

Failure Modes and Recovery

Dispatch failures

If a CLI agent exits non-zero or returns empty output, the orchestrator records cli_error or empty_output. The run continues, metrics capture failed agents, and the smoke test target points operators at CLI flag mismatches.

Build/test regressions

Autofixes are considered suspect until build and tests pass against the pre-captured baseline. Regressions trigger bounded repair attempts before the round proceeds with a warning.

Context misalignment

Noise is not dropped silently. Every false positive becomes evidence for improving CLAUDE.md, PR descriptions, repo config, or the per-agent domain prompt set.

Cleanup failures

Cleanup is best effort. Residual worktrees do not break the next run because setup explicitly detects and reuses stale review worktrees.

Modification Guide

Change review scope

Modify domain prompts or repo-level disabled_domains, disabled_paths, and severity_overrides. This shifts what gets surfaced without changing orchestration flow.

Add new agent families

The clean seam is the dispatcher boundary in multi_review.py. Keep the output schema stable so classification, metrics, and summary generation do not need structural rewrites.

Tune autonomy safely

Mode detection, fix thresholds, retry counts, and round caps are the main control knobs. They should remain centralized so degraded behavior stays explainable.

Expand observability

Add fields to history JSON and the terminal metrics block together. Timing, counts, and failure reasons should be kept symmetric between console output and persisted artifacts.
