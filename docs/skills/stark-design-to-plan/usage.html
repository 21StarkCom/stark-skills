USAGE VIEW

stark-design-to-plan

Convert a finished design or spec into an execution-ready implementation plan with phases, dependencies, rollback steps, and risk controls.

This skill sits in the middle of the Stark workflow: review the design first, then generate competing implementation plans, cross-review them, and synthesize a stronger final plan before sending that plan into plan review.

When to Use It

Use this after a design doc is already written and reviewed. The input is a finished markdown spec. The output is not code, but a phased implementation plan you can execute or review.

What You Provide

A readable design file path, optional agent list, and optional controls for timeout, dry run, or forcing a run when the design file is dirty in Git.

What You Get Back

A final plan markdown file, a cross-review summary, terminal scorecards and metrics, optional PR posting, and raw history artifacts for auditability.

Outputs You Should Expect

Final plan markdown

If the design ends with -design.md, the plan becomes -plan.md. Otherwise the skill appends .plan.md.

Cross-review summary

A sibling file ending in .d2p-review.md containing the scorecard, strengths, weaknesses, and synthesis decisions.

Terminal progress

Task-based progress messages, timestamped phase logs, and a metrics block summarizing duration, review counts, winner, runner-up, and merge notes.

History folder

Raw generation JSON, cross-review JSON, synthesis markdown, and summary markdown saved under the design-to-plan history path for traceability.

Common Usage Patterns

Standard Reviewed Design

/stark-design-to-plan docs/specs/payments-design.md

Use when the design is settled and you want the normal full workflow: three draft plans, cross-review, synthesis, and output files.

Agent Comparison Run

/stark-design-to-plan docs/specs/payments-design.md --dry-run --timeout 900

Use when you want to inspect plan quality and scorecards without producing final files yet, especially for large or risky designs.

Dirty Working Copy Override

/stark-design-to-plan docs/specs/payments-design.md --force

Use only when the current uncommitted edits are intentional and the plan should reflect those edits rather than the last commit.

How the Winner Is Chosen

Scoring Model

Each plan is scored by the other agents on completeness, feasibility, phasing, risk, and testability. The average score determines the winner. If the top two are within 0.5 points, the skill treats it as a close race and synthesizes more cautiously.

Synthesis Behavior

The winning plan becomes the base. Better sections from other plans can be merged in, repeated weaknesses are removed, and concrete details such as file paths, commands, and function names are preserved instead of generalized.

The final synthesis is also quality-checked to ensure every design section maps to plan tasks and every phase includes rollback and verification.

Failure Handling
Failure What You See What To Do
No path provided Usage error immediately Provide a real markdown design path
File not found Candidate search in docs/ if the name looks partial Pick one of the candidates or pass the full path
Uncommitted changes Warning and abort unless forced Commit or stash the file, or rerun with --force
0 of 3 plan generations succeed Dispatch failure diagnostics Fix the environment or rerun after checking scripts and agents
1 of 3 plan generations succeed Warning, no cross-review Use the single plan or rerun for better comparison quality
Cross-review parse failure Raw review output fallback Review synthesis carefully; the skill may score manually from raw text
Script missing Setup error Run installation so the Stark scripts and Python venv exist
Prerequisites

Installed Tooling

The Stark scripts directory and its Python environment must exist. The skill expects the dispatch and GitHub helper scripts to be available from the configured scripts path.

Repository Context

The design file must be readable from the repo. Git is required for dirty-file checks. GitHub CLI is only needed if the run should post back to a pull request.

Plan Quality Dependency

This skill does not repair a weak design. If the design document is incomplete or contradictory, the final plan will inherit that ambiguity even after synthesis.
