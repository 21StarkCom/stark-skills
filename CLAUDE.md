## What This Is

Multi-agent PR code review system. Claude, Codex, and Gemini are all enabled (Gemini â†’ `gemini-3.1-pro-preview` on Vertex). The Vertex **project/location are resolved at runtime** by `tools/vertex_config_lib.ts` (env > config > `GOOGLE_CLOUD_PROJECT` > local `gcloud`) â€” **never hardcoded/committed in source**. Note: `-latest` aliases like `gemini-pro-latest` only resolve via the Generative-Language API-key fallback, **not** Vertex. Hierarchical config (global â†’ org â†’ repo). Self-improving prompts via review history analysis.

## Operating Principles

This is a **personal playground**, not production. No customers depend on it; the only user is the author.

- **No rollout ceremony.** Skip soaking, gating, smoking, canary, and gradual-rollout patterns. Merge straight to main once the PR is green.
- **Branch + PR for everything â€” nothing lost.** Every change lands on a branch and merges via PR; never commit or push to `main` directly. "Ship straight to main" means merge once green, not bypass the PR. This repo **is** the review system â€” so every review's findings get posted on the PR as comments (inline where anchored, summary otherwise; `multi_review.ts` already does the GitHub posting). Don't drop, downgrade, or summarize findings away, and don't merge with open findings unaddressed â€” fix them, or reply on the thread saying why not.
- **Draft PRs by default â€” test locally, un-draft to merge.** Every PR-opening path opens a **draft** so WIP stays out of draft-guarded CI while you verify locally. The single owner of the default is the `pr create` CLI (`tools/github_app.ts` â†’ `github_app_lib.ts::prCreate`, `draft ?? true`); the stark-gh `pr-open` tool, the review / red-team / spec-to-plan / phase-execute skills, and the automation-fleet prompts all inherit or set it. Opt out per-run with `--ready` (alias `--no-draft`). **You cannot merge a draft**, so the merge paths un-draft first: `/stark-gh:pr-merge` (and `github_app.ts pr ready` / `gh pr ready`) mark the PR ready-for-review â€” which fires the target CI via the `ready_for_review` event â€” then wait for green, then squash-merge. For "no CI on WIP" to actually hold, each target repo's `pull_request` workflows need the **skip-draft guard** (`if: github.event.pull_request.draft == false` + `ready_for_review` in `types`) â€” pattern + `== false` vs `!= true` guidance in `standards/workflows/skip-draft-guard.md`. Exempt: push-triggered publishers like `marketplace-sync` that must run CI + auto-merge (they open ready).
- **Language preference:** Go for backend, TypeScript for scripts. **Avoid Python at all costs** â€” the repo's tooling is now TypeScript-only (`tools/`); the former Python orchestrators + dispatch infra under `scripts/` were migrated out. Do not introduce new Python.
- **Test live.** Local-only verification is not enough. If a flow touches GCP, exercise the real GCP surface.
- **Always update documentation.** Any change that affects behavior, structure, commands, env vars, or operations must update the relevant docs (this file and `AGENTS.md` included) in the same change.

## Distribution

This repo is the **source of truth** for the skills + tools. Distribution is **marketplace-only** â€” the former `install.sh` symlink installer was removed (2026-07-07); there is no local symlink install step.

- **bifrost** (the marketplace; formerly `stark-marketplace`, renamed 2026-07-11) â€” the `bifrost` repo packages these skills as self-contained Claude Code plugins (`/plugin marketplace add 21StarkCom/bifrost`, then `/plugin install <bundle>@bifrost`). Its `catalog/` is **generated from this repo** by `stark sync`, and its engine vendors `tools/`+`global/` (config + prompts) into each plugin so an install needs no symlinks and no setup step. `/plugin update <bundle>@bifrost` pulls the latest published version. CI auto-publishes: `.github/workflows/marketplace-sync.yml` regenerates the marketplace and opens a PR on every push to `main` touching a vendored asset root â€” `skill/`, `tools/`, `global/`, `scripts/`, `standards/`, or `plugins/stark-gh/`.
- **Local dev is no longer live.** Editing a file in this repo does not take effect until the change is published (merge to `main` â†’ `marketplace-sync` PR â†’ merge) and the plugin is reinstalled/updated (`/plugin update â€¦`). To test an in-progress edit against a real install, run `stark sync` in the marketplace repo and `/plugin update` locally.
- **Assets plugins don't cover** (user `~/.claude/settings.json`, statusline, output-styles, `org/evinced` overrides, mutable-state dirs, git hooks) are now managed by hand â€” the installer no longer provisions them.

**The plugin-resolution seam:** `tools/asset_root_lib.ts` â€” `assetRoot()` resolves immutable assets (tools/prompts/config) from `${CLAUDE_PLUGIN_ROOT}` when running inside an installed plugin, falling back to `~/.claude/code-review` for direct (non-plugin) invocations such as automation-fleet crons; `stateRoot()` always stays under `$HOME`. Skills mirror this with `${CLAUDE_PLUGIN_ROOT:-$HOME/.claude/code-review}`. **Never hardcode `~/.claude/code-review/{tools,prompts,scripts,standards}` in a skill or tool** â€” route immutable-asset reads through `asset_root_lib`/the dual-fallback so plugin installs keep working.

### Pipeline (end-to-end, in order)

- `/stark-review-spec <path> [--fable] [--lead-agent codex|claude] [--lead-model ID]` â€” lead/wing spec review: codex (gpt-5.6-sol, xhigh) reviews 9 domains in parallel (cap 3 concurrent), claude (opus-4-8) wing applies JSON patches. Pass `--fable` (or `--lead-agent claude`) to run the **lead review on Fable 5** (`claude-fable-5`) â€” explicit opt-in; the wing/fixer stays opus-4-8. Multi-round fix loop with per-round git commit; growth/non-convergence circuit breakers stop pathological loops early; a net-reducing coherence pass (contradictions/repetitions/fluff/leftovers, `--no-coherence` to skip) runs before the final review-only round that captures unresolved; every run writes process analytics (health grade + `<doc>.review-analytics.md` sidecar). Tournament mode removed in the TS port. Delegates to `tools/stark_review_doc.ts`. Findings always land on a PR â€” **opens one (feature branch + push, authored by stark-claude) if none exists**; a spec under the retired `docs/superpowers/**` tree is relocated to `docs/specs/` on that branch first. **Every finding is posted as its own resolvable review thread, fixed, and resolved:** after dispatch, `tools/review_doc_findings.ts post` threads every finding onto the PR (auto-resolving the wing-fixed ones), then the skill fixes each still-open finding â€” **asking the operator via `AskUserQuestion` when a fix is ambiguous or has options** â€” commits + pushes, and resolves each thread with the fix; below-threshold findings are fixed too.
- `/stark-red-team-spec <path> [--source-spec PATH] [--model ID] [--dry-run] [--no-pr-comment] [--ready] [--fold]` â€” adversarial committee challenge of a spec doc (auto-opened PR is a **draft** by default; `--ready`/`--no-draft` opens it ready) (5 personas Ă— 1 round, default `gpt-5.5-pro`); writes `<spec>.red-team.md` sidecar and always lands findings on a PR â€” **opens one (pushing the spec doc + sidecar) if none exists**, else edits in place; the PR + findings comment are authored by the GitHub App matching the run's agent (`claude*`â†’stark-claude, `gpt*`/`o*`â†’stark-codex, `gemini*`â†’stark-gemini; codex by default); challenge-only, no fix loop (opt-in `--fold` runs `/stark-red-team-fold` after a successful challenge)
- `/stark-spec-to-plan <path>` â€” generate implementation plan from spec doc via paired lead/wing loop (default lead `claude`, wing `codex`); lead drafts, wing reviews and emits JSON verdict, fix-loop until approved. Cheaper and lower-variance than the prior 3-agent tournament. On approval the plan + per-round review summary **always land on a PR** â€” reuses an existing one, else **opens one** (branch + push, authored by the lead's GitHub App) â€” unless `--dry-run`. Generate/review/revise prompts (`global/prompts/spec-to-plan/`) carry the writing-plans authoring craft: per-task **Interfaces** (Consumes/Produces signatures), a **Global Constraints** header, task right-sizing, and a named proving **Test** for behavior-changing tasks, and an **SSOT** constraint (a task must consume an existing owner, never plan to hardcode a literal or re-derive a rule â€” shift-left prevention that complements the `ssot` review domain); the wing gates on the interface + test + SSOT items and the revise step self-reviews before emitting.
- `/stark-review-plan <path> [--fable] [--lead-agent codex|claude] [--lead-model ID]` â€” lead/wing execution plan review: codex (xhigh) reviews 5 adversarial domains (`completeness`, `security`, `sequencing`, `viability`, `ssot`), claude wing applies JSON patches. ## Conventions

- Prompts are per-agent: each LLM gets its own version of each domain
- Domain IDs are slugs derived from filenames: `01-architecture.md` â†’ `architecture`
- Config uses JSON, prompts use markdown
- Agent preambles in `agent.md`, domain prompts in `NN-domain.md`
- **Every skill honors `--help`.** Each `skill/*/SKILL.md` opens with a `## Help` block right after its frontmatter that points at the shared `standards/help.md` protocol: a standalone `--help`/`-h`/`help` token in `$ARGUMENTS` prints the skill's purpose + usage + `## Arguments` and stops (no preflight, no phases, side-effect-free). The `skill_smoke_test.test.ts` guard asserts every skill references `standards/help.md`.
